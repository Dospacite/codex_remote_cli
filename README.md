# codex_remote_cli

`codex_remote_cli` is the bridge process for Codex Remote.

It runs next to your local `app-server`, enrolls a bridge identity with the public relay, prints a pairing code for the mobile app, and forwards encrypted Codex traffic between the relay and the local workstation.

## What it does

- starts a local `app-server` with a randomly generated local capability token
- enrolls or reuses a long-lived bridge identity
- prints a pairing code and ASCII QR for the mobile app
- connects to the relay over WebSocket
- forwards end-to-end encrypted Codex frames between mobile client and workstation
- serves relay-safe filesystem shims and relay-backed HTTP file downloads from the bridge host

## Requirements

- Python 3.12+
- an `app-server` binary available on `PATH`
- a reachable Codex Remote relay server

## Install

```bash
python3 -m venv .venv
. .venv/bin/activate
pip install -e .
```

## Quick start

Start the bridge and enroll it with a relay:

```bash
codex-remote-cli \
  --relay-url https://relay.example.com \
  --enroll-token your-bootstrap-token \
  --bridge-label "My workstation" \
  serve
```

On first run the CLI:

1. creates a long-lived bridge identity
2. enrolls that identity with the relay
3. stores the returned device ID and pairing code in the local config file
4. launches `app-server`
5. prints a one-time pairing code and QR
6. waits for the mobile app to connect

Once the mobile app claims the pairing code, the bridge keeps reconnecting to the relay and relaunching `app-server` if needed.

## Commands

### `serve`

Runs the bridge normally. This is the default command if you omit the subcommand.

```bash
codex-remote-cli --relay-url https://relay.example.com serve
```

### `pairing-code`

Prints the last pairing code and QR stored in the config file without starting the bridge.

```bash
codex-remote-cli --config ~/.config/codex_remote_cli/config.json pairing-code
```

This only works if the config already contains a pairing code from a previous successful enrollment.

## Configuration

CLI flags:

- `--config`: config file path, default `~/.config/codex_remote_cli/config.json`
- `--relay-url`: relay base URL, default `https://cr.rousoftware.com`
- `--bridge-label`: human-readable bridge name, default `My workstation`
- `--enroll-token`: optional bootstrap token required by the relay when enrollment protection is enabled
- `--local-port`: local `app-server` port, default `47123`
- `--ready-timeout`: seconds to wait for `app-server` readiness, default `30`
- `--app-server-bin`: `app-server` executable path or name, default `codex app-server`
- `--app-server-cwd`: working directory for the spawned `app-server`

Environment variables:

- `CODEX_REMOTE_RELAY_URL`
- `CODEX_REMOTE_BRIDGE_LABEL`
- `CODEX_REMOTE_ENROLL_TOKEN`
- `CODEX_REMOTE_LOCAL_PORT`
- `CODEX_REMOTE_READY_TIMEOUT`
- `CODEX_REMOTE_APP_SERVER_BIN`
- `CODEX_REMOTE_APP_SERVER_CWD`

## Local files

The bridge stores a JSON config file with:

- the relay URL
- the bridge label
- the enrolled device ID
- the bridge private key
- the bridge signing public key
- the latest pairing code

The config file is written with mode `0600`.

## Networking model

- Control plane: WebSocket to the relay at `GET /ws`
- Enrollment: `POST /api/v1/bridge/enroll`
- Relay downloads: temporary HTTPS URLs generated by the bridge and served through the relay
- Local app-server: private loopback WebSocket secured with a capability bearer token

For relay mode, the CLI does not expose your workstation directly to the internet. The relay sees authenticated bridge traffic, but Codex session payloads are encrypted end to end between the mobile client and the bridge.

## Security model

- bridge identities use long-lived Ed25519 signing keys
- per-session transport keys use X25519 key agreement and ChaCha20-Poly1305
- local `app-server` access is protected by a random capability token written to a temporary file
- pairing depends on relay-issued codes, not a fixed shared secret
- relay-backed HTTP downloads use short-lived, bridge-scoped tokens rather than direct file paths
